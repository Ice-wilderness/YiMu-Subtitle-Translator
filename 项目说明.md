```
翻译脚本/
```

**config.json 说明：**
- 首次启动时不存在，程序使用 `app.py` 中 `DEFAULT_CONFIG` 的默认值（均为空）

- 包含字段：`analysis_*`（分析模型 API）、`translation_*`（翻译模型 API）、`batch_size`、`parallel_batches`、`default_prompt`、`whisper_*`（转录引擎）、`min_chinese_chars`（默认5）、`min_duration`（默认0.5）、`merge_max_gap`（默认0.5）、`dedup_min_overlap`（默认8）
- 已加入 `.gitignore`，不会随代码上传至 GitHub

## 二、整体架构

**纯前端 SPA + 轻量 Flask 后端**

- 数据全部在浏览器内存（`AppState` 对象），刷新即清空
- 后端职责：① 代理调用 AI API（避免跨域）；② 读写 config.json；③ 本地 Whisper 转录
- 4 个 Tab 是同一个页面的不同 `<section>`，通过 `display:none/block` 切换
- 没有数据库，没有用户系统

> 版本更新历史见 [CHANGELOG.md](CHANGELOG.md)

## 五、各 Tab 逻辑

### Tab 1 — 导入文件
- 支持 SRT 文件直接上传、音视频文件本地 Whisper 转录
- `handleFileUpload(file)` → `parseSRT(text)` → 存入 `AppState.subtitles`

### Tab 2 — 短句合并
- 按语言类型（CJK/Latin）自动合并碎片字幕
- 直接修改 `AppState.subtitles`（不可撤销）

### Tab 3 — AI 矫正 & 翻译

**Phase 1 分析矫正：**
1. `runAIAnalysis()` → POST `/api/analyze` → AI 返回误识别词 JSON
2. `renderAIPreview()` 渲染卡片，用户确认后 `commitCorrections()` 应用

**Phase 2 翻译（两种模式）：**

- **最终翻译**：`startFinalTranslation()` → POST `/api/translate`（异步任务，并行翻译）
- **一键翻译（推荐）**：`startOneclickTranslation()` → POST `/api/oneclick`，自动执行：
  1. 内容分析 → 生成术语表 + 领域识别
  2. 并行翻译 → 术语表注入每批 prompt
  3. 质量检测 → 中文字数 + 持续时间（纯规则，无 AI）
  4. AI 智能修复 → 见「七、一键翻译流程详解」

**翻译 Prompt 格式：**
```
序号 | 英文原文
1 | This design process...
```
要求输出：
```
序号 | 英文原文 | 中文译文
1 | This design process... | 设计师一直被教导的...
```

**对齐算法（`_align`）：** 翻译 API 返回的文本需要和原始批次对应，按三级优先级匹配：

| 优先级 | 策略 | 触发条件 | 说明 |
|--------|------|----------|------|
| 1 | **序号匹配** | AI 返回的有效序号 ≥ 批次数量的 80% | 按序号一一对应，缺失的条目返回空字符串 |
| 2 | **顺序匹配** | 返回行数与批次数量完全相等 | 按位置顺序对应，不依赖序号 |
| 3 | **原文归一化匹配** | 以上两种均不满足时兜底 | 将 AI 返回的原文列（第二列）归一化后与源条目原文做字典匹配 |

正常情况下走策略1。策略2和3是应对模型未按格式返回序号时的降级保障。序号匹配缺失的条目译文为空，后续会被质量检测标记。

### Tab 4 — 导出

**上半部分 — 位置调换（可选）：**
- 选择双语上下排列顺序，选择后立即生效

**下半部分 — 预览下载：**
- 预览最终字幕，下载 SRT 文件

## 七、一键翻译流程详解

```
POST /api/oneclick → 返回 task_id
前端每 1.5s 轮询 GET /api/oneclick/status/<task_id>

阶段 1: analyzing   → AI 内容分析（1 次 API 调用）→ 术语表 + 领域
阶段 2: translating  → 并行翻译（N 批并发）→ 术语表注入每批 prompt
阶段 2.5: dedup     → 去重修复（纯程序，无 API，< 1ms）
                       检测 N+1 条译文是否以 N 条译文为前缀（重复拼接）
                       条件：N 条译文 >= dedup_min_overlap 字（默认8）
                       修复：直接裁掉 N+1 条开头的重复部分
阶段 3: checking     → 后端质量检测（纯计算，无 API）
                       条件：中文字数 < min_chinese_chars（默认5）
                          OR 持续时间 < min_duration（默认0.5s）
阶段 4: fixing       → AI 智能修复（仅问题条目 + 上下文）
                       · AI 只判断方向：merge_up / merge_down / skip
                       · 拦截层1：两条间隔 > merge_max_gap（默认0.5s）强制skip
                       · 拦截层2：already_merged 集合防止重复合并
                       · 工程合并：软件拼接原文+时间轴 → 重新调翻译 API 得译文
完成:   done         → 返回翻译结果 + 统计信息
```

**进度上报字段：**
```json
{
  "status": "running",
  "stage": "translating",
  "progress": 60,
  "message": "正在翻译第 3/5 批...",
  "analysis": { "domain": "科技", "glossary": [...] }
}
```

## 八、AI 修复设计原则

### 为什么 AI 只给方向，不生成译文？

早期方案让 AI 直接输出合并后的 `new_translation`，结果 AI 在生成译文时会"自然续写"，把上一条的内容也带进来，产生重复拼接（幻觉）。

新方案的分工：
- **AI 负责语义判断**：仅决定 merge_up / merge_down / skip，判断两条字幕是否语义连贯、该向哪个方向合并
- **软件负责工程操作**：拼接原文、合并时间轴、调翻译 API 重新得到译文

这样 AI 的输出只是一个方向枚举值，没有内容生成，彻底消除幻觉来源。

### 为什么需要去重修复（阶段 2.5）？

翻译模型天然倾向于输出"语义完整的句子"。遇到被切断的半句话时，模型可能把下一条的内容提前补全进当前条的译文里，同时下一条又完整翻译了自己——形成"N+1 条译文以 N 条译文为前缀"的重复。

这类问题特征明确，用字符串前缀匹配就能检测，无需 AI，放在质量检测之前处理掉，能减少后续 AI 修复的误判压力。

### 为什么需要时间间隔拦截和 already_merged？

- **时间间隔拦截**：两条字幕之间间隔较大（> 500ms）通常意味着说话人切换或段落边界，强制跳过防止跨说话人合并
- **already_merged**：相邻两条同时被检测为短句时，AI 可能对两条都给出合并指令，导致同一对条目被合并两次。用集合记录已参与合并的条目，确保每条只参与一次合并

## 十一、事件系统

只有一个自定义事件：`subtitlesChanged`

- **触发时机：** 任何修改 `AppState.subtitles` 的操作完成后
- **监听方：** Tab 4（导出页）监听此事件自动刷新；字幕编辑器同步渲染
